# ds-siga — Design System SIGA

Repositório do Storybook do Design System SIGA, publicado em GitHub Pages via workflow automático.

## Regras obrigatórias para stories (`.stories.ts`)

### 1. `applicationConfig` em todo story
Todo `meta` deve ter o decorator `applicationConfig`. Sem ele, o componente não renderiza no GitHub Pages (Storybook Angular 10.x + componentes standalone).

```typescript
import { applicationConfig } from '@storybook/angular';
import { provideRouter } from '@angular/router';

const meta: Meta<MeuComponente> = {
  title: 'Componentes/MeuComponente',
  component: MeuComponente,
  tags: ['autodocs'],
  decorators: [applicationConfig({ providers: [provideRouter([])] })],
  ...
};
```

### 2. `moduleMetadata` obrigatório em `render()` com template
Quando usar `render()` com `template:`, declarar o componente explicitamente:

```typescript
render: (args) => ({
  props: { ...args },
  template: `<siga-meu-componente [prop]="prop"></siga-meu-componente>`,
  moduleMetadata: { imports: [MeuComponente] },
}),
```

### 3. Títulos únicos
Nunca criar dois stories com o mesmo `title`. Causa conflito silencioso — um dos stories some do Storybook.

### 4. Caminhos de assets relativos
Não usar caminhos absolutos para assets (ex: `/logo.png`). O GitHub Pages serve de `/ds-siga/`, então `/logo.png` aponta para o lugar errado. Usar `'logo.png'` (sem barra).

### 5. Stories de protótipo vs componente real
- `src/lib/components/*/` — stories dos componentes reais da biblioteca
- `src/stories/components/` — protótipos inline (prefixar title com `'Protótipos/'`)
- `src/stories/foundations/` — tokens de design (prefixar title com `'SIGA/'`)

## Deploy

Push na branch `main` → workflow `.github/workflows/deploy-storybook.yml` executa `npm run build-storybook` e publica automaticamente. Nenhuma etapa manual necessária.
